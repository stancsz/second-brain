# G04 — WIP chain: rename `drawer` → `Concept`

## Goal
Close R4: the code and docs agree on OKF terminology. No `drawer` /
`drawers` / `Drawer` / `DRAWER` in any production source, test, hook,
CLI, or user-facing doc. SQLite table name `drawers` migrates to
`concepts`; function names `_drawer_to_md` / `_row_to_drawer` /
`_import_drawers` migrate to `_concept_to_md` / `_row_to_concept` /
`_import_concepts`; CLI subcommand "drawer" renames to "concept"; CLI
arg `--drawer` renames to `--concept`. The migration is a SQLite
`ALTER TABLE drawers RENAME TO concepts` (SQLite 3.25+ supports this
natively) plus column-rename where needed.

## Why a chain
G04 is the highest-effort single gap in the active backlog. The
drawer→Concept identifier appears ~400 times across scripts/, tests/,
hooks/, references/, commands/, docs/. A single-iteration refactor
risks:
- mechanical substitutions that miss context (e.g., "drawer" in a
  string literal that documents the historical name)
- regressions in the SQLite schema that the existing verifiers
  (G01-G03, G19, G20) might not catch
- doc drift (the rename needs to land in docs/02 and references/ in
  the same iter as the code)

The chain splits the work by **invariant boundary**: each milestone
ships a verifier that proves its slice is clean. The full corpus
stays green at the end of every milestone, so a partial milestone
ship is still shippable.
